# 📄 Timekeeping System Plan: Master Blueprint (v3.1 - Payroll Grade)

## 1. Project Overview
A highly accurate, fingerprint-based timekeeping system designed for payroll integration. It automatically calculates workdays using overlap-time logic, cleanly separates Standard vs. Overtime (OT) boundaries, and maintains strict idempotency and audit trails for manual corrections.

---

## 2. Business Rules & Operational Boundaries

### 2.1. Time Blocks & Strict Boundaries
To prevent overlap and double-counting, time is strictly partitioned:
* **Core Working Block:** 08:00 - 17:30 (Max 8 working hours).
* **Grace Period:** 17:30 - 18:30 (Checkout allowed without penalty; adds no extra standard credit if 8 hours are already met).
* **Absolute OT Block:** 18:30 - 24:00 (Requires an approved OT request).
* **Lunch Break (Overlap based):** 12:00 - 13:30.
* **Dinner Break (Overlap based):** 18:30 - 20:00 (1.5 hours deducted if working OT).

### 2.2. Check-in/Check-out & Sync Logic
* **Timezone & Drift:** All timestamps must be synced and stored in UTC/Server Timezone.
* **Midnight Crossing (Late Checkout):** If a scan occurs between 00:00 and 04:00 AM without a previous check-out, it is assigned to the previous calendar day.
* **Missing Scans:** No checkout = 0 credit. Requires an approved `Explanation Request` to trigger recalculation.

### 2.3. Leaves, Holidays & Weekends
* Standard daily credit defaults to 0 on weekends and official holidays unless a specific shift is assigned.
* Approved Leaves (Paid Time Off) insert a virtual log mapping to 8 hours of standard credit, bypassing raw log calculation.

### 2.4. Manual Correction (Explanations)
* Applied when an employee forgets to Check-in or Check-out.
* **Limit:** Maximum 2 times/month/employee.

### 2.5. Authentication & Fingerprint Onboarding
* **Account Provisioning:** HR/Manager must first create an employee profile in the Core System (generating an `emp_id`).
* **Fingerprint Registration:** The employee physically registers their fingerprint at one of the master fingerprint devices.
* **ID Mapping:** The unique ID generated by the fingerprint machine must be mapped to the `fingerprint_id` field in the `Employees` table by HR. Without this mapping, the system will discard the employee's `CheckinLogs` as "Unknown Device ID".

## 3. Mathematical Models & Overlap Logic (Core Engine)

To resolve the "negative time" and "fixed break deduction" risks, the system uses an Overlap Function. All times are evaluated in **Seconds**.

$$Overlap(A_{start}, A_{end}, B_{start}, B_{end}) = \max(0, \min(A_{end}, B_{end}) - \max(A_{start}, B_{start}))$$

### 3.1. Standard Work Calculation (Capped at 18:30)
To prevent standard work logic from bleeding into the OT block, we cap the effective check-out time for standard calculation at `18:30:00`.

$$T_{out\_std} = \min(T_{out}, \text{18:30:00})$$

$$S_{lunch\_overlap} = Overlap(T_{in}, T_{out\_std}, \text{12:00:00}, \text{13:30:00})$$

$$S_{work} = (T_{out\_std} - T_{in}) - S_{lunch\_overlap}$$

$$Daily\_Credit = \min\left(1.0, \frac{S_{work}}{28800}\right)$$

### 3.2. Overtime (OT) Calculation
OT strictly begins at `18:30:00`. 
If $T_{out} \le \text{18:30:00}$, then $OT\_Credit = 0$. Otherwise, establish the OT start boundary and deduct the overlapping dinner time.

$$OT_{in} = \max(T_{in}, \text{18:30:00})$$

$$S_{dinner\_overlap} = Overlap(OT_{in}, T_{out}, \text{18:30:00}, \text{20:00:00})$$

$$S_{OT\_actual} = (T_{out} - OT_{in}) - S_{dinner\_overlap}$$

$$OT\_Credit = \frac{\min(S_{OT\_actual}, S_{OT\_registered})}{28800}$$

## 4. Payroll-Grade Database Schema

### 4.1. `Employees` (Core Identity)
* `emp_id` (PK)
* `fingerprint_id` (UNIQUE): Mapped to the hardware device.
* `account_username` (UNIQUE), `hourly_rate`.

### 4.2. `CheckinLogs` (Immutable Raw Data)
* `log_id` (PK)
* `emp_id`, `device_id`, `timestamp`
* `sync_hash` (UNIQUE): Hash of `emp_id + timestamp + device_id` to guarantee idempotency.

### 4.3. `DailyAttendance` (Processed & Auditable)
* `attendance_id` (PK)
* `emp_id`, `date`
* `standard_credit`, `ot_credit`
* `calc_status`: PENDING, CALCULATED, RECALCULATED, ERROR.
* `formula_version`: e.g., "v3.1".
* `audit_trail`: JSON column storing historical changes.

### 4.4. `Requests` (Explanations, OT, Leaves)
* `req_id` (PK), `emp_id`, `type` (OT, EXPLANATION, LEAVE).
* `date`, `status` (PENDING, APPROVED, REJECTED).
* **OT Specific Fields:** `start_time`, `end_time`, `registered_seconds` (Used as $S_{OT\_registered}$ in the formula).

### 4.5. `MonthlyLimits` (Concurrency Control)
To prevent race conditions when checking the "max 2 explanations/month" rule during concurrent API calls:
* `limit_id` (PK), `emp_id`, `month_year`.
* `explanation_count`: Integer.
* **Constraint:** `CHECK (explanation_count <= 2)`. Updates via atomic increments or Pessimistic Locking (`SELECT ... FOR UPDATE`).
