# 📄 Timekeeping System Plan: Master Blueprint (v3.1 - Payroll Grade)

## 1. Project Overview
A highly accurate, fingerprint-based timekeeping system designed for payroll integration. It automatically calculates workdays using overlap-time logic, cleanly separates Standard vs. Overtime (OT) boundaries, and maintains strict idempotency and audit trails for manual corrections.

---

## 2. Business Rules & Operational Boundaries

### 2.1. Time Blocks & Strict Boundaries
To prevent overlap and double-counting, time is strictly partitioned:
* **Core Working Block:** 08:00 - 17:30 (Max 8 working hours).
* **Grace Period:** 17:30 - 18:30 (Checkout allowed without penalty; adds no extra standard credit if 8 hours are already met).
* **Absolute OT Block:** 18:30 - 24:00 (Requires an approved OT request).
* **Lunch Break (Overlap based):** 12:00 - 13:30.
* **Dinner Break (Overlap based):** 18:30 - 20:00 (1.5 hours deducted if working OT).

### 2.2. Check-in/Check-out & Sync Logic
* **Timezone & Drift:** All timestamps must be synced and stored in UTC/Server Timezone.
* **Midnight Crossing (Late Checkout):** If a scan occurs between 00:00 and 04:00 AM without a previous check-out, it is assigned to the previous calendar day.
* **Missing Scans:** No checkout = 0 credit. Requires an approved `Explanation Request` to trigger recalculation.

### 2.3. Leaves, Holidays & Weekends
* Standard daily credit defaults to 0 on weekends and official holidays unless a specific shift is assigned.
* Approved Leaves (Paid Time Off) insert a virtual log mapping to 8 hours of standard credit, bypassing raw log calculation.

### 2.4. Manual Correction (Explanations)
* Applied when an employee forgets to Check-in or Check-out.
* **Limit:** Maximum 2 times/month/employee.

### 2.5. Authentication & Fingerprint Onboarding
* **Account Provisioning:** HR/Manager must first create an employee profile in the Core System (generating an `emp_id`).
* **Fingerprint Registration:** The employee physically registers their fingerprint at one of the master fingerprint devices.
* **ID Mapping:** The unique ID generated by the fingerprint machine must be mapped to the `fingerprint_id` field in the `Employees` table by HR. Without this mapping, the system will discard the employee's `CheckinLogs` as "Unknown Device ID".

## 3. Mathematical Models & Overlap Logic (Core Engine)

To resolve the "negative time" and "fixed break deduction" risks, the system uses an Overlap Function. All times are evaluated in **Seconds**.

Overlap(A_start, A_end, B_start, B_end) = max(0, min(A_end, B_end) - max(A_start, B_start))

### 3.1. Standard Work Calculation (Capped at 18:30)
To prevent standard work logic from bleeding into the OT block, we cap the effective check-out time for standard calculation at `18:30:00`.

T_out_std = min(T_out, 18:30:00)
S_lunch_overlap = Overlap(T_in, T_out_std, 12:00:00, 13:30:00)
S_work = (T_out_std - T_in) - S_lunch_overlap
Daily_Credit = min(1.0, S_work / 28800)

### 3.2. Overtime (OT) Calculation
OT strictly begins at `18:30:00`. 
If T_out <= 18:30:00, then OT_Credit = 0. Otherwise, establish the OT start boundary and deduct the overlapping dinner time.

OT_in = max(T_in, 18:30:00)
S_dinner_overlap = Overlap(OT_in, T_out, 18:30:00, 20:00:00)
S_OT_actual = (T_out - OT_in) - S_dinner_overlap
OT_Credit = min(S_OT_actual, S_OT_registered) / 28800

## 4. Payroll-Grade Database Schema

### 4.0. Global Enums
* **CalcStatus:** `PENDING`, `CALCULATED`, `RECALCULATED`, `ERROR`
* **RequestType:** `OT`, `EXPLANATION`, `LEAVE`
* **RequestStatus:** `PENDING`, `APPROVED`, `REJECTED`

### 4.1. `Employees` (Core Identity)
* `emp_id`: INT (PK, Auto-increment)
* `fingerprint_id`: VARCHAR (UNIQUE) - Mapped to the hardware device.
* `account_username`: VARCHAR (UNIQUE)
* `password_hash`: VARCHAR [note: 'Argon2 or bcrypt hash for login security']
* `full_name`: VARCHAR
* `hourly_rate`: DECIMAL(10,2)
* `created_at`: TIMESTAMP (Default: now)

### 4.2. `CheckinLogs` (Immutable Raw Data)
* `log_id`: BIGINT (PK, Auto-increment)
* `emp_id`: INT (FK)
* `device_id`: INT
* `timestamp`: DATETIME
* `sync_hash`: VARCHAR(255) (UNIQUE) - Hash of `emp_id + timestamp + device_id` to guarantee idempotency.
* `is_used`: BOOLEAN (Default: false)

### 4.3. `DailyAttendance` (Processed & Auditable)
* `attendance_id`: BIGINT (PK, Auto-increment)
* `emp_id`: INT (FK)
* `date`: DATE
* `standard_credit`: DECIMAL(5,4)
* `ot_credit`: DECIMAL(5,4)
* `calc_status`: ENUM (`CalcStatus`)
* `formula_version`: VARCHAR(10) - e.g., "v3.1".
* `audit_trail`: JSON - Stores history of manual overrides.
* `updated_at`: TIMESTAMP (Default: now)

### 4.4. `Requests` (Explanations, OT, Leaves)
* `req_id`: INT (PK, Auto-increment)
* `emp_id`: INT (FK)
* `type`: ENUM (`RequestType`)
* `date`: DATE
* `status`: ENUM (`RequestStatus`)
* `start_time`: TIME (Nullable)
* `end_time`: TIME (Nullable)
* `registered_seconds`: INT (Nullable) - Used as S_OT_registered in the formula.
* `created_at`: TIMESTAMP (Default: now)

### 4.5. `MonthlyLimits` (Concurrency Control)
To prevent race conditions when checking the "max 2 explanations/month" rule during concurrent API calls:
* `limit_id`: INT (PK, Auto-increment)
* `emp_id`: INT (FK)
* `month_year`: VARCHAR(7) - Format: YYYY-MM.
* `explanation_count`: INT
* **Constraint:** `CHECK (explanation_count <= 2)`. Updates via atomic increments or Pessimistic Locking (`SELECT ... FOR UPDATE`).

## 5. System Architecture & Use Cases (Text Flow)

### 5.1. System Actors
* **Employee:** Standard system user submitting requests and scanning fingerprints.
* **Manager / HR:** Admin user managing accounts, approving requests, and exporting payroll.
* **Fingerprint Device (External):** Hardware module capturing templates and pushing attendance logs.
* **Time / Scheduler (Trigger):** Automated system Cron jobs running daily overlap calculations.

### 5.2. Use Case Modules & Interactions

**Module 0: Authentication & Onboarding**
* **UC0a: Login:** Both *Employee* and *Manager/HR* authenticate into the system.
* **UC0b: Provision Account & Enroll Fingerprint:** *Manager/HR* creates the account. The *Employee* provides physical fingerprint data. The *Fingerprint Device* captures the template and assigns a hardware ID.

**Module 1: Ingestion & Processing**
* **UC1: Scan Fingerprint:** *Employee* interacts with the *Fingerprint Device*. The device identifies the employee.
* **UC2: Sync Logs (Idempotency):** The *Fingerprint Device* pushes log data to the system API. (This action inherently triggers *UC3* logic or schedules it).
* **UC3: Calculate Overlap:** The *Time/Scheduler* triggers the daily calculation engine processing synced logs.

**Module 2: Request Management**
* **UC4: Submit OT Request:** Initiated by *Employee*.
* **UC5: Submit Explanation (Max 2/month):** Initiated by *Employee*.
* **UC6: Approve / Reject:** *Manager/HR* reviews UC4 and UC5 requests. (Approving a request inherently includes *UC7*).
* **UC7: Recalculate Credit & Audit Log:** Triggered automatically upon request approval.

**Module 3: Reporting & Payroll**
* **UC8: View Dashboard:** Accessed by both *Employee* (personal view) and *Manager/HR* (global view).
* **UC9: Freeze & Export:** *Manager/HR* finalizes the data period for payroll integration.

## 6. Implementation Roadmap & QA Phase
* Phase 1: Onboarding & Ingestion: Build the UI and API for HR to create accounts, map fingerprint_id, and handle user authentication. Set up machine sync and implement the sync_hash logic for idempotency.
* Phase 2: Core Calculation Engine: Implement the Overlap Math formulas in the Backend.
* Phase 3: Event-Driven Overrides & Concurrency: Build the Request module with Pessimistic Locking for limits. Implement recalculation triggers on approval.
* Phase 4: QA, Test Matrix & UAT: Generate mock datasets covering edge cases (Half-days, late clock-ins at 12:45, partial OT). Payroll UAT: Compare system output against HR's manual Excel calculation over a 2-week period.
* Phase 5: Reporting & Export: Dashboard construction and final payroll export locking (Freezing data).
